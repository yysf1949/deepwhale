## Theme 1: Multi-Agent Safety

A multi-agent system is one where a single user-facing agent spawns
one or more sub-agents (each with their own tool execution context,
their own audit-log partition, and their own capability restrictions).
v5.0 production hardening covers per-instance events (D-96..D-104)
but does NOT cover sub-agent isolation, capability-based delegation,
or rollback to a "pre-sub-agent" state. v6.0 closes this gap.

**First sub-sprint (D-107 candidate):** Multi-agent safety policy minimal
seed. See "First sub-sprint" section below for full design.

**Status:** Planning preview → executable after D-106.

## Theme 2: Hosted/Enterprise Opt-In Gates

Hosted deployments (cloud-managed deepwhale instances) need additional
gates that the default registry does not enforce: per-tenant rate
limits, billing/quota checks, SSO/OIDC integration, audit-log shipping
to a third-party SIEM, and per-tenant data-residency policies. v5.0
has none of these; the user-facing code path is single-tenant.

**First sub-sprint (D-NN candidate):** Per-tenant rate-limit middleware.
The middleware enforces a `RateLimitPolicy { tenantId, maxReqPerMin, ... }`
and returns 429 when the limit is exceeded. Future D-NN+ can layer
billing/quota, SSO, and SIEM on top of the same middleware.

**Status:** Planning preview (no entry sub-sprint yet; will be picked
after multi-agent safety seed is implemented).

## Theme 3: Distributed Cross-Instance Coordination

A distributed deepwhale deployment runs multiple instances in different
regions or availability zones. They need to coordinate: leader
election (one instance is "active", others are "standby"), distributed
audit-log aggregation (all instances ship events to a central
collector), conflict-free replicated state for shared config (e.g.
the DistributionManifest is the same across all instances).

v5.0 production hardening covers single-instance events (D-104
cross-instance is about reading the prior instance's log on the
next start, not coordinating live distributed instances).

**First sub-sprint (D-NN candidate):** Simple leader election via a
shared lock file (NFS or S3-style) or via Redis. The leader
performs writes; the standby polls and takes over if the leader's
lease expires.

**Status:** Planning preview.

## Theme 4: Advanced Observability

v5.0 has audit logging (D-87/88/89/90/102) but no tracing, no
metrics, no alerting, no dashboards. v6.0 adds the standard 4
pillars of observability:

1. **Tracing**: each tool call is a span; spans are correlated
   via a `traceId`; spans are exported to OpenTelemetry-compatible
   backends (Jaeger, Tempo, etc.).
2. **Metrics**: counters and histograms for tool calls, error
   rates, latency p50/p95/p99, audit-event rates, etc.
3. **Alerting**: rule-based alerts (e.g. "alert if tool error rate
   > 5% in 5min"); alerts are emitted as AuditLog events so
   downstream tooling can route them.
4. **Dashboards**: pre-built Grafana dashboards for the 3 most
   common views (per-tool usage, per-tenant usage, error rates).

**First sub-sprint (D-NN candidate):** OpenTelemetry-compatible
span export. Reuse D-89 PersistingAuditLog as the span sink;
add a `recordSpan(span)` method to AuditLog.

**Status:** Planning preview.

## First sub-sprint: Multi-Agent Safety Policy Seed

This is the first concrete v6.0 sub-sprint (D-107 candidate). It
extends the v5.0 plugin-governance theme (D-91..D-103) and the
v5.0 production-hardening theme (D-96..D-104) to cover sub-agents.

### Goals

1. **Sub-agent identity**: every sub-agent has a `SubAgentId`
   (branded string, similar to `ToolName`).
2. **Sub-agent capability**: sub-agents declare a subset of the
   v5.0 `ToolCapability` vocabulary (e.g. a `summarizer` sub-agent
   may only declare `file-read` + `code-execute`; it cannot declare
   `shell-exec` or `network`).
3. **Sub-agent audit partition**: every audit event emitted by a
   sub-agent is tagged with `subAgentId`. Reuse D-89 PersistingAuditLog
   by adding a `subAgentId?` field to `AuditEvent`.
4. **Sub-agent rollback**: when a sub-agent fails or exceeds its
   capability, the parent can `rollbackSubAgent(subAgentId)` which
   marks the sub-agent's last N events as `rolled-back` and emits
   a `sub-agent-rollback` event.
