## Theme 1: Multi-Agent Safety

A multi-agent system is one where a single user-facing agent spawns one or
more sub-agents, each with its own task boundary, capability restrictions,
and audit/rollback context. v5.0 production hardening covers per-instance
events; v6.0 closes the sub-agent isolation and rollback gap.

**Seed status:** Complete through D-110.

Shipped files:

- `packages/coding-agent/src/multi-agent/sub-agent.ts`: `SubAgentId`,
  `SubAgent`, `SubAgentRegistry`, and helper guards (D-107).
- `packages/coding-agent/src/multi-agent/sub-agent-policy.ts`:
  `enforceSubAgentPolicy`, reusing D-103 profile policy enforcement (D-108).
- `packages/coding-agent/src/multi-agent/rollback-sub-agent.ts`:
  `rollbackSubAgent` for dry-run/apply rollback events (D-109).
- `packages/coding-agent/src/multi-agent/sub-agent-policy-snapshot.ts`:
  `buildSubAgentPolicySnapshot` bridge over policy and rollback (D-110).

Remaining Theme 1 work:

- Sub-agent-to-sub-agent communication.
- Sub-agent runtime ownership and lifecycle cleanup.
- Cross-thread or distributed sub-agent scheduling.

## Theme 2: Hosted/Enterprise Opt-In Gates

Hosted deployments need gates that the default local registry does not
enforce: per-tenant rate limits, billing/quota checks, SSO/OIDC validation,
audit-log shipping to SIEM, and per-tenant data-residency policy.

**Seed status:** Started and seed part 3 of 3+ complete through D-113.

Shipped files:

- `packages/coding-agent/src/enterprise/tenant-rate-limit.ts`:
  per-tenant rate limiting (D-111).
- `packages/coding-agent/src/enterprise/tenant-quota.ts`:
  per-tenant billing/quota accounting (D-112).
- `packages/coding-agent/src/enterprise/sso-oidc.ts`:
  claim-level SSO/OIDC validation without JWT signature verification (D-113).

Remaining Theme 2 work:

- SIEM export.
- Data-residency policy.
- Hosted deployment wiring.
- Theme 2 cross-bridge that ties rate limit, quota, SSO, and SIEM state
  into one policy snapshot.

## Theme 3: Distributed Cross-Instance Coordination

A distributed deepwhale deployment runs multiple instances in different
regions or availability zones. They need leader election, distributed
audit-log aggregation, and replicated shared configuration.

**Status:** Planned, not implemented.

First candidate sprint:

- Simple leader election via a shared lease.
- Standby takeover when the active lease expires.
- Audit events that distinguish active, standby, failover, and conflict
  resolution states.

## Theme 4: Advanced Observability

v5.0 has audit logging, dump/query support, fatal event formatting, and
rollback decisions. v6.0 adds production observability primitives:

1. Tracing: each tool call becomes a span correlated by trace id.
2. Metrics: counters and histograms for tool calls, errors, latency, and
   audit-event rates.
3. Alerting: rule-based alerts emitted as audit events.
4. Dashboards: standard views for tool usage, tenant usage, and error rates.

**Status:** Planned, not implemented.

First candidate sprint:

- OpenTelemetry-compatible span export that can reuse the D-89/D-90 audit
  persistence and query pattern.
