## 3 个 user-reported bug 修完

### Bug 1: `/pro` DeepSeek reasoning_content 400 (commit 90db74a)

**现象**: deepwhale `/pro` (任何 DeepSeek V4 多轮) 第二轮报 `HTTP 400 "The 'reasoning_content' in the thinking mode must be passed back to the API"`.

**根因**: Sprint 1a "机制 3 简化" 拍板主动丢 `reasoning_content` (parse.ts:45-46). DeepSeek V4 默认开 thinking mode, 多轮必须回传上轮 reasoning_content, 否则 400. 1b/1c 时代都没补这个简化.

**修法** (5 文件 + 1 tool-loop + 1 session-adapter + 3 unit tests):
- types.ts: `ChatMessage` / `ChatResult` / `ChatChunk.delta` 加 `reasoning_content?: string`
- parse.ts: parseOaiChatCompletion + parseSseEvent 提取 reasoning_content
- deepseek-client.ts: stream 累加 + toWireMessage 透传 (omitempty style)
- session/jsonl.ts: SessionEvent.assistant 加 reasoning_content? 字段 (向后兼容)
- session-adapter.ts: toolLoopStepToSessionEvent 写 + sessionEventsToMessages 读 双向
- tool-loop.ts: assistant message 推 working 带 reasoning_content (核心 fix)

### Bug 2: 默认走 Anthropic, 不是 DeepSeek-first (commit 99aa314)

**现象**: user 装 Claude Code 后 ANTHROPIC_AUTH_TOKEN 自动设, 又设了 DEEPSEEK_API_KEY, 期望 DeepSeek 走, 实际 Anthropic 走. 更糟, "Both set" 错被 tryCreateClient catch 静默, stderr 啥都不说, REPL 仍能进, user 不知道 chat 没在工作.

**根因** (3 层):
1. `llm-factory.ts:85-86` if-else 顺序 "hasAnthropic 排第一" 跟 README "DeepSeek-first" 矛盾
2. `tryCreateClient` catch 静默存 clientError, stderr 不输出
3. 没"user 决断"路径 (两个都设只能 unset 之一, 不友好)

**修法** (1 文件 + 1 repl.ts + 4 unit tests):
- 调换 if-else 顺序: hasDeepseek 排第一 (跟品牌一致)
- `tryCreateClient` catch 时 stderr 写 `[deepwhale] init error: <msg>` (dedup'd)
- 新 env `DEEPWHALE_PROVIDER=<deepseek|anthropic>` 让 user shell 一行 set 决断
  - 非法值 (e.g. typo) 静默 fall through, 不 fatal
  - 提示: `DEEPWHALE_PROVIDER=deepseek deepwhale` 不用 unset 另一个 key
- Both-set 错信息更新, 提示 3 个修法 (unset / DEEPWHALE_PROVIDER / --provider)

### Bug 3: cache_hit_rate 96%↔85% footer 跳变焦虑 (commit 99aa314 EMA 部分)

**现象**: footer `cache: 90%` per-turn 显示, 实测 96% → 50% → 96% → 50% → 96% 跳变 (tool result 长 → prompt 涨 → ratio 跌). user 以为 cache 失效.

**根因** (短期 vs 长期):
- 短期 (本次修): per-turn 数字是真值, 但没 trend 上下文, user 焦虑
- 长期 (Sprint 2+ backlog): compaction 改 "append summary 不 replace head" 让 DeepSeek server-side prefix cache key 不变. 风险大, 跟 D-5+ 已 ship 协议不兼容, 不动.

**修法** (短期, EMA 平滑):
- 新增 `UsageEmaState { hitRateEMA?, sampleCount }` 在 startRepl 闭包
- `formatUsageStatus` 接受 emaState, 显示 `cache: 90% (avg 85%)` (per-turn 真值 + trend)
- `appendUsageStatus` in-place 更新 EMA: `newEMA = 0.5 * current + 0.5 * oldEMA`
  - α=0.5, 5-turn half-life, 5-turn 前数据权重 3.1% (基本忘掉)
- 旧 caller 不传 emaState = 默认 EMPTY_EMA, 行为兼容 (不显示 avg 段)
- sampleCount < 3 不显示 avg (样本太少趋势不稳)
- reload session → emaState reset → avg 段消失, user 看到就知道 reload 过了 (visible feedback, not silent)
